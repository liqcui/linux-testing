# LMbench结果解读示例

本文档提供典型的测试结果和详细解读，帮助理解性能数据。

## 1. 内存带宽测试解读

### 典型输出示例

```
Memory Copy Bandwidth:
  Size      Bandwidth    Cache Level      解读
  ----      ---------    -----------      ----
  512B      12.5 GB/s    L1              ← 峰值带宽（L1缓存内）
   1KB      12.8 GB/s    L1              ← 最大带宽点
   2KB      12.6 GB/s    L1              ← 仍在L1内，略有波动
   4KB      11.2 GB/s    L1              ← 4KB后下降，受缓存行影响
   8KB      10.5 GB/s    L1
  16KB       8.5 GB/s    L1边缘          ← 开始溢出L1
  32KB       6.2 GB/s    L2              ← 超出L1缓存（典型32KB）
  64KB       5.8 GB/s    L2
 128KB       5.5 GB/s    L2              ← 接近L2缓存带宽
 256KB       4.9 GB/s    L2边缘          ← 开始溢出L2
 512KB       4.2 GB/s    L3              ← 超出L2缓存（典型256KB）
   1MB       3.8 GB/s    L3
   2MB       3.5 GB/s    L3
   4MB       3.2 GB/s    L3              ← 接近L3缓存带宽
   8MB       3.0 GB/s    L3边缘          ← 开始溢出L3
  16MB       2.8 GB/s    内存            ← 超出L3缓存（典型8MB）
  32MB       2.6 GB/s    内存            ← 接近主内存带宽
  64MB       2.5 GB/s    内存            ← 稳定在内存带宽
```

### 关键指标解读

#### 1. 带宽平台区识别

**什么是带宽平台区？**
在特定大小范围内，带宽值保持相对稳定，这个范围称为"带宽平台区"。每个缓存层级都有自己的平台区。

**示例分析：**
```
L1平台（512B-16KB）:
  512B: 12.5 GB/s
   1KB: 12.8 GB/s  ← 峰值
   2KB: 12.6 GB/s
   4KB: 11.2 GB/s
   8KB: 10.5 GB/s
  平均: ~12 GB/s，波动约15%

L2平台（32KB-256KB）:
  32KB: 6.2 GB/s
  64KB: 5.8 GB/s
 128KB: 5.5 GB/s
  平均: ~5.8 GB/s，波动约12%

L3平台（512KB-8MB）:
 512KB: 4.2 GB/s
   1MB: 3.8 GB/s
   2MB: 3.5 GB/s
   4MB: 3.2 GB/s
  平均: ~3.7 GB/s，波动约14%

内存平台（>16MB）:
  16MB: 2.8 GB/s
  32MB: 2.6 GB/s
  64MB: 2.5 GB/s
  平均: ~2.6 GB/s，波动约6%
```

**正常标准：**
- 平台区内波动 < 20%: ✅ 正常
- 平台区内波动 20-30%: ⚠️ 需要关注
- 平台区内波动 > 30%: ❌ 异常，需要调查

#### 2. 缓存边界识别

**什么是缓存边界？**
当数据大小超过某级缓存容量时，带宽会明显下降，这个下降点就是缓存边界。

**示例分析：**
```
L1→L2边界（16KB → 32KB）:
  16KB: 8.5 GB/s
  32KB: 6.2 GB/s
  下降: (8.5-6.2)/8.5 = 27%
  ✅ 正常范围（20-50%）

L2→L3边界（256KB → 512KB）:
 256KB: 4.9 GB/s
 512KB: 4.2 GB/s
  下降: (4.9-4.2)/4.9 = 14%
  ✅ 正常范围（20-30%）- 渐进式下降

L3→内存边界（8MB → 16MB）:
   8MB: 3.0 GB/s
  16MB: 2.8 GB/s
  下降: (3.0-2.8)/3.0 = 7%
  ⚠️ 下降偏小，可能是渐进式过渡
```

**缓存大小推断：**
从边界点可以推断缓存大小：
- L1 缓存: 约 16-32 KB（每核心）
- L2 缓存: 约 256-512 KB（每核心）
- L3 缓存: 约 8-16 MB（共享）

#### 3. 异常模式识别

**模式A: 锯齿波动（异常）**
```
❌ 异常示例:
   1KB: 12 GB/s
   2KB: 8 GB/s   ← 异常下降 33%
   4KB: 11 GB/s  ← 回升
   8KB: 7 GB/s   ← 再次下降
  16KB: 10 GB/s  ← 再次回升

可能原因:
  1. NUMA远程访问
  2. 缓存冲突/抖动
  3. 内存控制器调度问题
  4. 测试干扰（系统负载）

诊断步骤:
  numactl --hardware              # 检查NUMA配置
  numastat                        # 检查NUMA统计
  mpstat 1                        # 检查CPU使用
  sar -r 1                        # 检查内存压力
```

**模式B: 无明显平台区（异常）**
```
❌ 异常示例:
  512B: 10 GB/s
   1KB:  9 GB/s   ← 持续下降
   2KB:  8 GB/s   ← 无稳定平台
   4KB:  7 GB/s
   8KB:  6 GB/s
  16KB:  5 GB/s

可能原因:
  1. 系统负载过高
  2. 测试受其他进程干扰
  3. CPU频率不稳定（温控限制）
  4. 内存带宽被其他进程占用

诊断步骤:
  top -b -n 1 | head -20         # 检查负载
  cpupower frequency-info        # 检查CPU频率
  ps aux --sort=-%mem | head     # 检查内存占用
```

**模式C: 意外高带宽（异常）**
```
❌ 异常示例:
  64MB: 15 GB/s  ← 超过L1带宽！

可能原因:
  1. 测量错误
  2. 编译器优化过度（循环被优化掉）
  3. 预取优化效果异常好
  4. 工具错误

诊断步骤:
  # 重新测试
  # 检查程序是否正确编译
  # 使用不同工具验证（如STREAM）
```

## 2. 内存延迟测试解读

### 典型输出示例

```
Random Access Latency (stride=64B):
  Size      Latency     Cache Level    解读
  ----      -------     -----------    ----
   4KB      1.2 ns      L1            ← 最低延迟（L1缓存）
   8KB      1.3 ns      L1            ← 延迟稳定，仍在L1
  16KB      1.5 ns      L1
  32KB      1.8 ns      L1边缘        ← 开始增加
  64KB      4.5 ns      L2            ← L1→L2 明显跳跃（2.5x）
 128KB      5.2 ns      L2            ← L2内延迟稳定
 256KB      5.8 ns      L2
 512KB      6.5 ns      L2边缘
   1MB     18.5 ns      L3            ← L2→L3 明显跳跃（2.8x）
   2MB     20.3 ns      L3            ← L3内延迟稳定
   4MB     22.1 ns      L3
   8MB     24.8 ns      L3边缘
  16MB     65.3 ns      内存          ← L3→内存 明显跳跃（2.6x）
  32MB     72.1 ns      内存          ← 内存延迟稳定
  64MB     75.8 ns      内存
 128MB     78.5 ns      内存
```

### 关键指标解读

#### 1. 延迟稳定区

**CPU周期计算：**
假设CPU频率 3.0 GHz = 3000 MHz = 0.33 ns/周期

```
L1延迟（1.2-1.8 ns）:
  1.2 ns = 1.2 / 0.33 ≈ 3.6 周期
  1.8 ns = 1.8 / 0.33 ≈ 5.4 周期
  ✅ 典型L1延迟: 3-6周期

L2延迟（4.5-6.5 ns）:
  4.5 ns = 4.5 / 0.33 ≈ 13.6 周期
  6.5 ns = 6.5 / 0.33 ≈ 19.7 周期
  ✅ 典型L2延迟: 12-20周期

L3延迟（18.5-24.8 ns）:
  18.5 ns = 18.5 / 0.33 ≈ 56 周期
  24.8 ns = 24.8 / 0.33 ≈ 75 周期
  ✅ 典型L3延迟: 50-80周期

内存延迟（65.3-78.5 ns）:
  65.3 ns = 65.3 / 0.33 ≈ 198 周期
  78.5 ns = 78.5 / 0.33 ≈ 238 周期
  ✅ 典型内存延迟: 180-250周期
```

#### 2. 延迟跳跃分析

**正常跳跃倍数：**
```
L1→L2跳跃:
  1.8 ns → 4.5 ns
  倍数: 4.5 / 1.8 = 2.5x
  ✅ 正常范围（2-4x）

L2→L3跳跃:
  6.5 ns → 18.5 ns
  倍数: 18.5 / 6.5 = 2.8x
  ✅ 正常范围（2-4x）

L3→内存跳跃:
  24.8 ns → 65.3 ns
  倍数: 65.3 / 24.8 = 2.6x
  ✅ 正常范围（2-3x）
```

**异常跳跃示例：**
```
❌ 跳跃过小:
L2→L3: 6.5 ns → 8.0 ns（1.2x）
  问题: L3访问未明显增加
  原因: 可能是预取掩盖了延迟

❌ 跳跃过大:
L2→L3: 6.5 ns → 45.0 ns（6.9x）
  问题: L3延迟异常高
  原因: 可能是NUMA远程访问
```

#### 3. Stride影响对比

**同一数据大小，不同stride：**
```
64KB测试，不同stride:
  Stride=16B:  4.0 ns  (基准)
  Stride=32B:  4.2 ns  (+5%)   一半缓存行
  Stride=64B:  4.5 ns  (+12%)  完整缓存行
  Stride=128B: 5.5 ns  (+37%)  跨缓存行
  Stride=256B: 7.2 ns  (+80%)  跨多个缓存行

结论:
  - Stride ≤ 缓存行: 延迟增加 < 15%
  - Stride = 缓存行: 延迟增加 10-20%
  - Stride > 缓存行: 延迟显著增加 > 30%
```

## 3. 上下文切换测试解读

### 典型输出示例

```
Process Context Switch Latency:
  Processes  DataSize  Latency     解读
  ---------  --------  -------     ----
  2          0B        2.3 us      ← 基准最低延迟
  2          64B       2.5 us      ← 数据量小，影响不大 (+9%)
  2          512B      3.1 us      ← 开始影响缓存 (+35%)
  2          1KB       3.8 us      ← 缓存污染增加 (+65%)
  2          4KB       5.2 us      ← 显著缓存污染 (+126%)

  4          0B        3.2 us      ← 进程数增加 (+39%)
  4          1KB       4.8 us      ← 进程+数据双重影响 (+109%)

  8          0B        4.5 us      ← 接近核心数
  8          1KB       6.5 us

  16         0B        6.8 us      ← 超过核心数 (+196%)
  16         1KB       10.2 us

  32         0B        12.5 us     ← 大量调度开销 (+443%)
  64         0B        25.3 us     ← 极高调度开销 (+1000%)
```

### 关键指标解读

#### 1. 基准延迟评估

**性能等级划分：**
```
优秀:  < 2.0 us  (现代高性能CPU，优化内核)
良好:  2-3 us    (主流配置)
一般:  3-5 us    (虚拟化环境、老旧硬件)
较差:  > 5 us    (性能问题，需要调查)
```

**不同环境的典型值：**
```
物理机（现代CPU）:        1.5-2.5 us
KVM虚拟机:               2.5-4.0 us
容器（Docker/K8s）:      2.0-3.5 us
公有云虚拟机:             3.0-5.0 us
```

#### 2. 进程数影响曲线

**理想进程数计算：**
```
假设8核CPU:

进程数 | 延迟 | 增长率 | 评价
-------|------|--------|------
  2    | 2.3  | 基准   | ✅ 最优
  4    | 3.2  | +39%   | ✅ 良好
  8    | 4.5  | +96%   | ✅ 可接受（=核心数）
  16   | 6.8  | +196%  | ⚠️ 开始退化
  32   | 12.5 | +443%  | ❌ 严重退化
  64   | 25.3 | +1000% | ❌ 不推荐

推荐: 进程数 ≤ CPU核心数 × 1.5
```

#### 3. 数据量影响分析

**缓存污染效应：**
```
2进程情况下，数据量影响:
  0B:   2.3 us  (基准，无缓存污染)
  64B:  2.5 us  (+9%)   一个缓存行
  512B: 3.1 us  (+35%)  8个缓存行
  1KB:  3.8 us  (+65%)  16个缓存行，L1污染
  4KB:  5.2 us  (+126%) 64个缓存行，严重L1污染

结论:
  - 每个缓存行增加 ~1% 延迟
  - 超过L1大小后，延迟急剧增加
  - 实际应用应尽量减少进程间传输数据量
```

## 4. 系统调用延迟解读

### 典型输出示例

```
System Call Latency:
  调用类型           延迟        分类          解读
  --------          ----        ----          ----
  getpid()          0.05 us     vDSO优化      ← 最快（无内核陷入）
  gettimeofday()    0.06 us     vDSO优化
  clock_gettime()   0.05 us     vDSO优化

  getppid()         0.15 us     简单调用      ← 快速查表
  getuid()          0.16 us     简单调用
  getgid()          0.17 us     简单调用

  read(0 bytes)     0.45 us     I/O调用       ← 无实际I/O
  write(1 byte)     0.68 us     I/O调用

  stat()            1.2 us      文件系统      ← 文件元数据查找
  fstat()           0.9 us      文件系统
  open/close        2.3 us      文件系统      ← 涉及inode操作

  fork()           85.0 us      进程管理      ← 创建子进程
```

### 关键指标解读

#### 1. vDSO优化调用

**什么是vDSO？**
vDSO（virtual Dynamic Shared Object）是一种内核优化技术，将某些简单的系统调用实现在用户态，避免内核态切换开销。

**性能对比：**
```
无vDSO优化:
  getpid(): 0.3-0.5 us  (需要内核态切换)

vDSO优化:
  getpid(): 0.05-0.08 us  (用户态完成)

性能提升: 4-8x
```

**检查vDSO是否启用：**
```bash
# 检查进程内存映射
cat /proc/self/maps | grep vdso
# 应该看到类似:
# 7fff12345000-7fff12346000 r-xp 00000000 00:00 0  [vdso]

# 如果没有，可能是内核配置问题
```

#### 2. KPTI/安全缓解影响

**性能对比：**
```
禁用KPTI:
  getppid():     0.15 us
  open/close:    1.8 us
  read/write:    0.5 us

启用KPTI:
  getppid():     0.35 us  (+133%)
  open/close:    3.2 us   (+78%)
  read/write:    0.9 us   (+80%)

结论:
  KPTI通常增加 50-150% 系统调用延迟
  vDSO优化的调用不受影响
```

**检查安全缓解状态：**
```bash
cat /sys/devices/system/cpu/vulnerabilities/*

# 示例输出:
# Meltdown: Mitigation: PTI  ← KPTI已启用
# Spectre v1: Mitigation: usercopy/swapgs barriers
# Spectre v2: Mitigation: Full AMD retpoline
```

## 使用建议

### 快速检查清单

测试完成后，按以下顺序检查：

1. **内存带宽曲线**
   - [ ] 是否有明显的带宽平台区？
   - [ ] 缓存边界是否清晰？
   - [ ] 最终内存带宽是否合理（>10 GB/s）？

2. **内存延迟曲线**
   - [ ] L1延迟是否 < 3 ns？
   - [ ] L2延迟是否 < 10 ns？
   - [ ] L3延迟是否 < 40 ns？
   - [ ] 内存延迟是否 < 100 ns？

3. **上下文切换**
   - [ ] 2进程/0字节延迟是否 < 4 us？
   - [ ] 进程数增加时延迟是否合理增长？

4. **系统调用**
   - [ ] vDSO调用是否 < 0.1 us？
   - [ ] 简单调用是否 < 0.5 us？
